SPIEGAZIONE CREAZIONE TABELLE MARIADB - FICSIT RESTAURANT OS
=============================================================

Questo file spiega come andrebbero create le tabelle MariaDB per sostituire in futuro il salvataggio attuale in LocalStorage.
Il sito non deve dipendere direttamente da MariaDB: la connessione al database dovra passare da un backend/API, mentre il frontend continuera a usare le stesse funzioni del data layer.


1. DATABASE PRINCIPALE
---------------------

Nome consigliato:

ficsit_restaurant

Impostazioni consigliate:

- CHARACTER SET: utf8mb4
- COLLATION: utf8mb4_unicode_ci

Motivo:
utf8mb4 permette di salvare correttamente testi con caratteri speciali, accenti, simboli e contenuti inseriti dagli utenti.

Esempio:

CREATE DATABASE IF NOT EXISTS ficsit_restaurant
  CHARACTER SET utf8mb4
  COLLATE utf8mb4_unicode_ci;

USE ficsit_restaurant;


2. TABELLA users
----------------

Scopo:
Salvare i dati dei clienti che effettuano una prenotazione.

Campi consigliati:

- id
  Identificativo numerico progressivo del cliente.
  Tipo: BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY

- nome
  Nome del cliente.
  Tipo: VARCHAR(80) NOT NULL

- cognome
  Cognome del cliente.
  Tipo: VARCHAR(80) NOT NULL

- email
  Email del cliente.
  Tipo: VARCHAR(160) NOT NULL
  Vincolo consigliato: UNIQUE

- telefono
  Numero di telefono.
  Tipo: VARCHAR(40) NOT NULL

- created_at
  Data di creazione del record.
  Tipo: TIMESTAMP DEFAULT CURRENT_TIMESTAMP

Nota:
L'email puo essere unica per evitare duplicati inutili, ma in fase di implementazione bisognera decidere se aggiornare un cliente gia esistente oppure crearne uno nuovo.


3. TABELLA tables
-----------------

Scopo:
Salvare i tavoli disponibili nella sala del ristorante.

Campi consigliati:

- id
  Identificativo numerico progressivo del tavolo.
  Tipo: BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY

- code
  Codice leggibile usato dal sito, ad esempio TABLE-HUB-01.
  Tipo: VARCHAR(40) NOT NULL UNIQUE

- capacity
  Numero massimo di persone.
  Tipo: INT NOT NULL

- status
  Stato del tavolo.
  Tipo: ENUM('libero', 'prenotato', 'occupato') NOT NULL DEFAULT 'libero'

- position_x
  Posizione orizzontale nella mappa.
  Tipo: DECIMAL(8,2) NOT NULL DEFAULT 0

- position_y
  Posizione verticale nella mappa.
  Tipo: DECIMAL(8,2) NOT NULL DEFAULT 0

- position_z
  Eventuale profondita o livello grafico.
  Tipo: DECIMAL(8,2) NOT NULL DEFAULT 0

Nota:
Il campo code e importante perche viene usato anche nelle prenotazioni come riferimento al tavolo.


4. TABELLA reservations
-----------------------

Scopo:
Salvare le prenotazioni effettuate dai clienti.

Campi consigliati:

- id
  Identificativo testuale della prenotazione, ad esempio RES-XXXX.
  Tipo: VARCHAR(40) PRIMARY KEY

- user_id
  Collegamento opzionale al cliente nella tabella users.
  Tipo: BIGINT UNSIGNED NULL
  Foreign key: REFERENCES users(id)

- table_code
  Codice del tavolo prenotato.
  Tipo: VARCHAR(40) NOT NULL
  Foreign key: REFERENCES tables(code)

- persons
  Numero di persone.
  Tipo: INT NOT NULL

- date
  Data della prenotazione.
  Tipo: DATE NOT NULL

- time
  Ora della prenotazione.
  Tipo: TIME NOT NULL

- status
  Stato della prenotazione.
  Tipo: ENUM('In attesa', 'Approvata', 'Rifiutata') NOT NULL DEFAULT 'In attesa'

- created_at
  Data di creazione.
  Tipo: TIMESTAMP DEFAULT CURRENT_TIMESTAMP

- updated_at
  Ultima modifica.
  Tipo: TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP

Relazioni:

- reservations.user_id -> users.id
- reservations.table_code -> tables.code

Nota:
Una prenotazione puo nascere con stato "In attesa" e poi essere approvata o rifiutata dall'area admin.


5. TABELLA menu
---------------

Scopo:
Salvare i piatti visibili nel menu cliente e gestibili dall'admin.

Campi consigliati:

- id
  Identificativo testuale del piatto, ad esempio MENU-001.
  Tipo: VARCHAR(40) PRIMARY KEY

- name
  Nome del piatto.
  Tipo: VARCHAR(120) NOT NULL

- category
  Categoria del piatto.
  Tipo: VARCHAR(80) NOT NULL

- description
  Descrizione del piatto.
  Tipo: TEXT NOT NULL

- price
  Prezzo del piatto.
  Tipo: DECIMAL(10,2) NOT NULL

- image
  Percorso o URL dell'immagine del piatto.
  Tipo: VARCHAR(255) NOT NULL

- available
  Indica se il piatto deve essere mostrato al cliente.
  Tipo: BOOLEAN NOT NULL DEFAULT TRUE

- created_at
  Data di creazione.
  Tipo: TIMESTAMP DEFAULT CURRENT_TIMESTAMP

- updated_at
  Ultima modifica.
  Tipo: TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP

Nota:
Per immagini caricate dall'admin e meglio salvare nel database il percorso del file, non il file pesante in formato base64.
I file immagine andrebbero salvati in una cartella del server oppure in uno storage dedicato.


6. TABELLA orders
-----------------

Scopo:
Salvare gli ordini effettuati dal cliente.

Campi consigliati:

- id
  Identificativo testuale dell'ordine, ad esempio ORD-XXXX.
  Tipo: VARCHAR(40) PRIMARY KEY

- reservation_id
  Collegamento opzionale a una prenotazione.
  Tipo: VARCHAR(40) NULL
  Foreign key: REFERENCES reservations(id)

- customer_name
  Nome del cliente che effettua l'ordine.
  Tipo: VARCHAR(160) NOT NULL

- items_json
  Lista dei piatti ordinati con quantita, prezzo e nome.
  Tipo: JSON NOT NULL

- total
  Totale dell'ordine.
  Tipo: DECIMAL(10,2) NOT NULL

- status
  Stato dell'ordine.
  Tipo: ENUM('Ricevuto', 'Accettato', 'In preparazione', 'Pronto', 'Consegnato') NOT NULL DEFAULT 'Ricevuto'

- created_at
  Data di creazione dell'ordine.
  Tipo: TIMESTAMP DEFAULT CURRENT_TIMESTAMP

Relazioni:

- orders.reservation_id -> reservations.id

Nota:
items_json e una scelta pratica per mantenere lo storico esatto dell'ordine anche se in futuro un piatto cambia nome o prezzo.
In una versione piu normalizzata si potrebbero creare anche le tabelle order_items e collegarle a menu.


7. TABELLA reviews
------------------

Scopo:
Salvare le recensioni inviate dai clienti.

Campi consigliati:

- id
  Identificativo testuale della recensione.
  Tipo: VARCHAR(40) PRIMARY KEY

- customer_name
  Nome del cliente.
  Tipo: VARCHAR(160) NOT NULL

- stars
  Valutazione da 1 a 5.
  Tipo: TINYINT NOT NULL
  Vincolo consigliato: CHECK (stars BETWEEN 1 AND 5)

- comment
  Testo della recensione.
  Tipo: TEXT NOT NULL

- created_at
  Data di invio recensione.
  Tipo: TIMESTAMP DEFAULT CURRENT_TIMESTAMP


8. ORDINE DI CREAZIONE CONSIGLIATO
----------------------------------

Le tabelle andrebbero create in questo ordine per rispettare le foreign key:

1. users
2. tables
3. menu
4. reservations
5. orders
6. reviews

Motivo:
reservations dipende da users e tables.
orders puo dipendere da reservations.
menu e reviews non hanno dipendenze obbligatorie da altre tabelle nello schema attuale.


9. INDICI CONSIGLIATI
---------------------

Oltre alle chiavi primarie e uniche, sono utili questi indici:

- users.email
  Per cercare rapidamente un cliente dalla email.

- tables.code
  Per collegare rapidamente prenotazioni e tavoli.

- reservations.date
  Per filtrare le prenotazioni per giorno.

- reservations.status
  Per mostrare rapidamente prenotazioni in attesa, approvate o rifiutate.

- orders.status
  Per filtrare gli ordini attivi nell'area admin.

- menu.category
  Per filtrare il menu lato cliente.

- menu.available
  Per mostrare solo i piatti disponibili.


10. COLLEGAMENTO FUTURO AL SITO
-------------------------------

Il sito attuale usa funzioni come:

- getMenu / saveMenu
- getReservations / saveReservations
- getOrders / saveOrders
- getReviews / saveReviews
- getDashboardStats

Quando MariaDB verra implementato, queste funzioni non dovranno piu leggere e scrivere LocalStorage.
Dovranno invece chiamare endpoint API del backend, ad esempio:

- GET /api/menu
- PUT /api/menu
- GET /api/reservations
- PUT /api/reservations
- GET /api/orders
- PUT /api/orders
- GET /api/reviews
- PUT /api/reviews
- GET /api/dashboard/stats

In questo modo il sito puo restare invariato lato HTML/CSS/JS delle pagine, mentre cambia solo il livello dati.


11. SICUREZZA CONSIGLIATA
-------------------------

Quando si passera a MariaDB reale:

- non inserire mai password database nel frontend;
- usare un backend per gestire le query;
- validare tutti i dati ricevuti dai form;
- proteggere l'area admin con login;
- usare HTTPS;
- usare un utente MariaDB con permessi limitati;
- fare backup periodici del database;
- evitare di salvare immagini grandi direttamente nel database.
